# 发布前置校验与版本事务设计

## 背景

当前 `release:publish` 在正式版流程中先修改版本、同步锁文件并创建 Git commit，随后才执行版本可用性检查、test、lint、build 和 pack。质量门禁失败时，仓库已经留下版本提交。

预发布流程虽然不会提交版本，但也会在质量门禁前临时改写目标包的 `package.json`。这会让发布准备动作和质量验证交织在一起，失败恢复依赖单一的 `EXIT trap`，且候选版本检查只能读取已被改写的文件。

发布流程需要统一为两个阶段：先执行不修改版本文件的不可变预检，全部通过后再进入版本事务和发布阶段。

## 目标

- 正式版和所有预发布通道在 test、lint、build、pack 等前置操作完成前不修改版本；
- 前置操作失败时不产生版本文件改动、Git commit、npm publish、Git tag 或 GitHub Release；
- 在不改写 `package.json` 的情况下计算候选版本并检查 npm 是否已存在；
- 正式版的版本写入和锁文件同步具备失败回滚能力；
- 预发布继续使用临时版本，并在成功或失败后恢复原始文件；
- 明确 npm 多包发布无法原子化时的停止和恢复边界；
- 用发布脚本测试固定命令顺序和失败路径。

## 非目标

- 不修改现有发布通道、发布目标或 CLI 参数；
- 不引入 Changesets、Lerna 等新的版本管理工具；
- 不改变 npm dist-tag、Git tag 或 GitHub Release 的命名规则；
- 不承诺多个 npm 包可以原子发布；npm registry 不提供跨包事务能力；
- 不使用独立 Git worktree 执行完整发布演练。

## 方案概述

采用“不可变预检 + 最终版本事务”方案。发布入口先解析目标包和候选版本计划，随后执行所有不会改写版本文件的检查。只有预检全部通过，才允许修改正式版本或写入临时预发布版本。

```text
解析发布参数
  → 生成候选版本计划
  → 校验分支、工作区与认证状态
  → 检查候选版本在 npm 上是否可用
  → test
  → lint
  → build
  → pack --dry-run
  → 预检通过
  → 写入正式或临时版本
  → 同步锁文件并提交（仅正式版）
  → npm publish
  → Git tag / push / GitHub Release（仅正式版）
```

## 候选版本计划

发布流程在内存中为每个目标包生成 `package@version` 计划，不提前修改文件。

### 正式版

- `current`：候选版本等于当前 `package.json` 版本；
- `patch`、`minor`、`major`：根据当前正式版本计算目标版本；
- `x.y.z`：单包发布时直接使用指定版本；
- `all` 继续禁止使用单个精确版本，避免强制不同包使用同一版本。

### 预发布

- `dev`、`alpha`、`beta`、`rc`、`next` 继续使用现有时间戳或唯一后缀规则；
- 候选版本只在计划阶段计算一次；
- npm 可用性检查、临时文件写入和实际 publish 必须使用同一个候选版本，避免时间变化导致版本不一致。

### 版本可用性检查

`check-versions-available.sh` 不再从目标包的 `package.json` 推断待发布版本，而是接收显式的包名和候选版本。检查失败时仍输出正式版的升级建议，但不得修改工作区。

## 不可变预检

### 通用检查

所有通道按以下顺序执行：

1. 校验发布参数和目标包；
2. 生成并展示候选版本计划；
3. 正式版校验 main 分支和完整工作区，预发布校验目标版本文件干净；
4. 校验 npm registry 和 npm 认证；
5. 正式版额外校验 GitHub CLI 认证；
6. 使用候选版本检查 npm 是否已存在；
7. 运行目标依赖链 test；
8. 运行目标依赖链 lint；
9. 构建目标依赖链；
10. 对目标发布包执行 `pack --dry-run`。

预检期间不得执行以下操作：

- `npm version`；
- 写入目标 `package.json`；
- `pnpm install --lockfile-only`；
- `git add` 或 `git commit`；
- `pnpm publish`；
- 创建或推送 Git tag；
- 创建 GitHub Release。

`pack --dry-run` 在预检阶段验证发布文件集合。候选版本的 semver 合法性和 npm 冲突由版本计划与显式可用性检查负责，避免为了检查包内容提前改写版本。

## 正式版本事务

预检通过后，正式版进入版本事务：

1. 备份目标包 `package.json` 和 `pnpm-lock.yaml`；
2. 按候选版本计划写入目标包版本；
3. 执行 `pnpm install --lockfile-only`；
4. 只暂存目标包版本文件和 `pnpm-lock.yaml`；
5. 创建版本提交；
6. 再次确认工作区干净；
7. 开始 npm publish；
8. 全部目标包发布成功后创建并推送 tag；
9. 创建 GitHub Release。

在 npm publish 开始前，版本写入、锁文件同步、暂存或 commit 任一步骤失败，都恢复备份并清理本次事务产生的暂存状态，不留下版本提交。

一旦 npm publish 已经开始，不自动撤销版本提交。发布命令可能已经在 registry 成功，但本地因网络中断返回失败；此时保留提交更利于核对和恢复。

## 预发布版本事务

预检通过后，预发布进入临时版本事务：

1. 备份目标包 `package.json`；
2. 写入计划阶段生成的候选预发布版本；
3. 注册 `EXIT trap`；
4. 按目标顺序执行带 dist-tag 的 npm publish；
5. 成功或失败后恢复所有目标包原始版本；
6. 清理备份目录和 trap。

预发布不修改锁文件、不创建版本 commit、不创建 Git tag，也不创建 GitHub Release。

## 多包发布边界

npm registry 不支持跨包原子事务。发布 `all` 时，如果前面的包已成功、后面的包失败，脚本无法撤回已经发布的版本。

脚本需要：

- 在发布前检查所有候选版本均可用；
- 在任何包失败后立即停止后续发布；
- 输出已经成功、尚未执行和失败的包；
- 正式版保留版本提交，供操作者重试未完成包或补建 tag；
- 预发布恢复本地临时版本，但明确提示 registry 可能已有部分版本；
- 不自动重试或覆盖已发布版本。

## 失败处理矩阵

| 失败阶段 | 修改版本文件 | 创建 commit | 开始 publish | 处理方式 |
| --- | --- | --- | --- | --- |
| 参数、分支、工作区或认证 | 否 | 否 | 否 | 直接退出 |
| 候选版本已存在 | 否 | 否 | 否 | 输出升级建议并退出 |
| test、lint、build 或 pack | 否 | 否 | 否 | 直接退出 |
| 正式版版本写入或 lockfile 同步 | 临时 | 否 | 否 | 恢复备份并退出 |
| 正式版 commit | 临时 | 否 | 否 | 恢复文件和暂存状态并退出 |
| npm publish 开始后 | 是 | 正式版为是 | 是 | 停止后续步骤并保留恢复现场 |
| tag、push 或 GitHub Release | 是 | 是 | 是 | 保留提交和已发布版本，输出补救提示 |
| 预发布 publish | 临时 | 否 | 是 | 恢复本地版本并报告可能的部分发布 |

## 测试设计

在 `scripts/release/release.test.sh` 中扩展 mock 失败注入和顺序断言。

### 顺序测试

- 正式版的 test、lint、build、pack 均早于 `npm version` 和 `git commit`；
- 预发布的 test、lint、build、pack 均早于临时版本写入；
- 候选版本可用性检查使用计划版本，而不是当前文件版本；
- 正式版 commit 早于 npm publish，tag 和 GitHub Release 晚于全部 publish。

### 失败路径测试

- test 失败时不修改版本、不执行 lockfile 同步、不 commit、不 publish；
- lint、build、pack 失败时遵循同一约束；
- 候选版本冲突时不进入质量门禁和版本事务；
- 正式版 lockfile 同步失败时恢复版本文件且不 commit；
- 正式版 commit 失败时恢复文件和暂存状态；
- 预发布 publish 失败时恢复原始版本；
- 多包 publish 中途失败时停止后续包，并输出部分发布状态。

### 回归测试

- 保留现有通道、目标、交互选择、认证和 tag 测试；
- 保留 `release:test` 只执行发布脚本测试的约束；
- 运行 `pnpm release:test`；
- 运行 shell 语法检查和仓库格式检查。

## 验收标准

- 任一质量门禁失败后，`git status` 与执行前一致；
- 任一质量门禁失败后，不存在新的版本 commit；
- 正式版和预发布均只在预检通过后写入候选版本；
- npm 可用性检查与实际 publish 使用完全相同的候选版本；
- 正式版版本事务在 publish 前失败时可以完整回滚；
- 预发布无论成功或失败都恢复原始版本文件；
- 多包部分发布时输出准确、可操作的恢复信息；
- 现有发布脚本测试与新增顺序、失败测试全部通过。
